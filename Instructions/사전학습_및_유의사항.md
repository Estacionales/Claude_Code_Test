# Claude Code 교육 — 사전 학습 가이드 및 유의사항

---

## 1. 개념 사전 학습

교육 효과를 높이기 위해 수강 전 아래 세 가지 개념을 미리 학습해 두세요. **총 30분 내외**면 충분합니다.

### 1-1. CLAUDE.md 작성법

- **무엇인가?** 프로젝트 루트에 두는 AI 행동 지침 파일로, Claude Code가 프로젝트 규칙을 인식하는 핵심 수단입니다.
- **어디서?** Claude Code 공식 문서 → "CLAUDE.md 작성법" 섹션을 일독하세요.
- **왜 중요한가?** Part 1(하네스 엔지니어링) 실습에서 직접 작성하므로, 파일 구조와 문법을 미리 파악해 두면 실습 시간이 크게 단축됩니다.

### 1-2. EARS 요구사항 문법

- **무엇인가?** 조건·동작·시스템 응답을 구조화된 문장으로 표현하는 요구사항 기술 방식입니다.
- **어디서?** 검색어: `"EARS requirements notation"` — 요약 1페이지 분량이면 충분합니다.
- **기본 구조 예시:**

  ```
  [WHEN 사용자가 로그인 버튼을 누르면]
  [WHILE 이메일 형식이 유효하지 않으면]
  [THE SYSTEM SHALL 오류 메시지를 표시해야 한다]
  ```

- **왜 중요한가?** Part 2(SDD) 전체가 EARS 문법 기반으로 진행됩니다. 문법이 낯설면 명세 작성에서 막혀 실습 진도가 늦어질 수 있습니다. 예시 문장을 3~5개 직접 써보는 것을 권장합니다.

### 1-3. Strangler Fig 패턴

- **무엇인가?** 레거시 시스템을 한 번에 교체하지 않고 모듈 단위로 점진적으로 신규 코드로 전환하는 아키텍처 패턴입니다.
- **어디서?** 마틴 파울러(Martin Fowler) 블로그의 "Strangler Fig Application" 포스트를 일독하세요.
- **핵심 흐름:**

  ```
  분석 → 격리(Facade 설정) → 이식(신규 모듈 작성) → 검증(구/신 동시 실행) → 절체(트래픽 이전)
  ```

- **왜 중요한가?** Part 4(레거시 전환) 전략의 핵심 개념입니다. 패턴의 단계별 의도를 이해하고 있어야 실습 중 의존성 분석 결과를 올바르게 해석할 수 있습니다.

## 2. 강의 진행 시 자주 막히는 포인트

각 파트별로 수강생이 실제로 자주 막히는 지점과 대비 방법을 정리했습니다. 사전에 확인하면 실습 흐름을 끊기지 않게 유지할 수 있습니다.

### Part 1. 하네스 엔지니어링

- **막히는 포인트:** MCP 서버 연결 오류
- **원인:** MCP 설정 파일(`settings.json`)의 JSON 문법 오류 또는 경로 설정 실수가 대부분입니다.
- **대비 방법:** MCP JSON 설정 문법(특히 `mcpServers` 블록 구조)을 미리 숙지하고, JSON 유효성 검사 도구(예: jsonlint.com)를 준비해 두세요.

### Part 2. SDD (Spec-Driven Development)

- **막히는 포인트:** EARS 문법이 어색하게 느껴짐
- **원인:** 자연어와 구조화된 명세 사이의 간극 — 처음엔 "언제(WHEN)", "조건(WHILE)", "동작(SHALL)"을 분리하는 것이 생소할 수 있습니다.
- **대비 방법:** 사전 학습 단계에서 본인 업무의 기능 하나를 골라 EARS 문장 3~5개를 직접 써보세요. 한 번 써보면 패턴이 자연스럽게 익혀집니다.

### Part 3. 실전 예제

- **막히는 포인트:** Java/Spring 환경 셋업 지연
- **원인:** 로컬 JDK 버전 불일치, Spring Boot 의존성 충돌 등 환경 문제가 실습 시간을 잡아먹습니다.
- **대비 방법:** 교육 전날 [Spring Initializr](https://start.spring.io)에서 빈 프로젝트를 생성해 빌드(`./mvnw package` 또는 `./gradlew build`)가 성공하는지 확인해 두세요.

### Part 4. 레거시 전환 전략

- **막히는 포인트:** 의존성 분석 결과 해석 어려움
- **원인:** Claude Code가 자동 생성한 클래스 다이어그램이나 의존성 맵을 읽는 것이 익숙하지 않으면 어떤 모듈을 먼저 교체해야 할지 판단이 서지 않습니다.
- **대비 방법:** UML 클래스 다이어그램(화살표 방향: 의존, 집합, 상속)의 기본 읽는 법을 간단히 복습해 두세요.

### Part 5. AI 거버넌스

- **막히는 포인트:** 법률 용어 생소함
- **원인:** AI 기본법 4대 의무(투명성·설명 가능성·안전성·책임성)의 개념이 낯설면 실습의 맥락을 이해하기 어렵습니다.
- **대비 방법:** AI 기본법 조문 요약본(1~2페이지 분량)을 미리 확인하고, 사내 AI 활용 정책 담당자가 누구인지 파악해 두세요.
